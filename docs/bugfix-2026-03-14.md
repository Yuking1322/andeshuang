## 一、严重 Bug 修复

### 1.1 浏览器端体检器下载崩溃（严重）

**问题描述**：
- 用户在页面上点击"下载一键体检器"按钮时，浏览器直接报错 `ReferenceError: Buffer is not defined`
- 根本原因：`src/utils/detectionScript.js:298` 的 `toUtf8BomBase64()` 函数使用了 Node.js 的 `Buffer.from()` 和 `Buffer.concat()`
- 该函数在两个地方被调用：
  - `scripts/generate-downloads.mjs`（Node 环境，正常）
  - `EnvironmentSelector.vue:316` 的 `handleDownloadDetection()`（浏览器环境，崩溃）

**修复方案**：
```javascript
// 修复前（Node.js 专用）
function toUtf8BomBase64(input) {
  const bom = Buffer.from([0xef, 0xbb, 0xbf])
  const body = Buffer.from(input, 'utf8')
  return Buffer.concat([bom, body]).toString('base64')
}

// 修复后（浏览器和 Node.js 通用）
function toUtf8BomBase64(input) {
  const encoder = new TextEncoder()
  const bom = new Uint8Array([0xef, 0xbb, 0xbf])
  const body = encoder.encode(input)
  const combined = new Uint8Array(bom.length + body.length)
  combined.set(bom)
  combined.set(body, bom.length)
  let binary = ''
  combined.forEach((byte) => {
    binary += String.fromCharCode(byte)
  })
  return btoa(binary)
}
```

**影响范围**：
- 修复前：用户无法在页面上动态下载体检器，只能下载预置的静态文件
- 修复后：浏览器端和 Node.js 端都能正常生成体检脚本

**相关文件**：
- `src/utils/detectionScript.js:298-310`

### 1.2 体检结果导入失败（严重）

**问题描述**：
- 用户运行体检脚本后，生成的 JSON 文件导回页面时解析失败
- 根本原因：PowerShell 5.1 的 `Set-Content -Encoding UTF8` 会在文件开头写入 BOM（`EF BB BF`），而 `JSON.parse()` 无法处理 BOM

**修复方案**：

**修复 1 — PowerShell 脚本改用无 BOM 写入**：
```javascript
// 修复前
$payload | ConvertTo-Json -Depth 6 | Set-Content -Path $outputPath -Encoding UTF8

// 修复后
$jsonText = $payload | ConvertTo-Json -Depth 6
[System.IO.File]::WriteAllText($outputPath, $jsonText, [System.Text.UTF8Encoding]::new($false))
```

**修复 2 — 前端导入时加 BOM 容错**：
```javascript
// 修复前
const payload = JSON.parse(await file.text())

// 修复后
const raw = await file.text()
const payload = JSON.parse(raw.replace(/^\uFEFF/, ''))
```

**影响范围**：
- 修复前：体检脚本能运行，JSON 能生成，但导回页面时报错
- 修复后：新生成的 JSON 无 BOM，旧的带 BOM 的 JSON 也能正常导入

**相关文件**：
- `src/utils/detectionScript.js:201-202`
- `src/components/EnvironmentSelector.vue:332-333`

### 1.3 预置包依赖顺序未排序（严重）

**问题描述**：
- `scripts/generate-downloads.mjs` 直接用 `preset.packageIds` 生成脚本，没有调用 `resolveDependencies()` 排序
- 虽然当前预置包碰巧手动排对了顺序（如 `['uv', 'python', 'pipx', 'jupyterlab']`），但如果有人改成 `['python', 'uv', ...]`，就会先装 Python（依赖 uv）再装 uv，导致失败
- 前端用户自选包时，`EnvironmentSelector.vue` 已经调用了 `resolveDependencies()`，所以前端没问题

**修复方案**：
```javascript
// 修复前
generateScript(preset.packageIds, {
  useChocolatey: true,
  mode: 'install',
  sourceLabel: 'Windows 预置包'
})

// 修复后
import { resolveDependencies } from '../src/data/environments.js'

generateScript(resolveDependencies(preset.packageIds), {
  useChocolatey: true,
  mode: 'install',
  sourceLabel: 'Windows 预置包'
})
```

**影响范围**：
- 修复前：预置包碰巧能用，但依赖顺序不稳定
- 修复后：预置包和前端动态生成都会自动排序依赖

**相关文件**：
- `scripts/generate-downloads.mjs:5,20`

## 二、安全性修复

### 2.1 会话签名验证未使用常量时间比较（中等）

**问题描述**：
- `functions/_shared/session.js:67` 使用普通字符串比较 `signature !== expectedSignature` 验证 HMAC 签名
- 虽然 HMAC 签名是 base64 编码的，时序攻击的实际风险较低，但同项目的 `local-auth.js` 已经正确使用了 `constantTimeEqual()`
- 为了安全一致性，应统一使用常量时间比较

**修复方案**：
```javascript
// 修复前
const expectedSignature = await signValue(encodedPayload, secret)
if (signature !== expectedSignature) return null

// 修复后
const expectedSignature = await signValue(encodedPayload, secret)
if (!constantTimeEqual(signature, expectedSignature)) return null

// 添加 constantTimeEqual 函数
function constantTimeEqual(left, right) {
  if (left.length !== right.length) return false

  let mismatch = 0
  for (let index = 0; index < left.length; index += 1) {
    mismatch |= left.charCodeAt(index) ^ right.charCodeAt(index)
  }

  return mismatch === 0
}
```

**影响范围**：
- 修复前：理论上存在时序攻击风险
- 修复后：与 `local-auth.js` 保持一致，消除潜在安全隐患

**相关文件**：
- `functions/_shared/session.js:67,119-128`

### 2.2 后端基础包混用 Chocolatey 和 winget 未检测（中等）

**问题描述**：
- `backend-starter.bat` 包含 OpenJDK（choco）、.NET SDK（winget）、Go（choco）
- 脚本会请求管理员权限并安装 Chocolatey，但没有检测 winget 是否存在
- 在 Windows LTSC 或精简版上，winget 可能不存在，导致 .NET SDK 安装失败

**修复方案**：
```javascript
// scriptGenerator.js 添加 winget 检测
const needsWinget = executablePackages.some((pkg) =>
  pkg.commands.some((command) => /\bwinget\b/i.test(command))
)

function buildWingetCheck() {
  return `echo [检查] winget 是否可用...
where winget >nul 2>nul
if %errorlevel% neq 0 (
  echo [警告] 未检测到 winget，某些软件可能无法安装。
  echo [提示] winget 在 Windows 10 1809+ 和 Windows 11 上默认可用。
  echo [提示] 如需安装 winget，请访问 Microsoft Store 搜索"应用安装程序"。
  echo.
  pause
  exit /b 1
) else (
  echo [跳过] 已检测到 winget。
)
echo.
`
}
```

**影响范围**：
- 修复前：在没有 winget 的系统上，.NET SDK 安装会静默失败
- 修复后：脚本会提前检测并给出明确提示

**相关文件**：
- `src/utils/scriptGenerator.js:70-73,270-287`

## 七、已知限制

### 7.1 日志文件名依赖系统日期格式
- `%DATE:~0,4%` 假设日期格式是 `YYYY-MM-DD` 或 `YYYY/MM/DD`
- 在美式区域设置（`MM/DD/YYYY`）下，日志文件名会错位
- 影响：日志文件名可能包含乱码，但不影响功能
- 建议：后续可改用 PowerShell 的 `Get-Date` 生成时间戳

### 7.2 深色模式未实现
- `colors.css` 定义了深色模式变量，但 `App.vue` 未使用
- 影响：用户在深色模式下仍然看到浅色界面
- 建议：后续可将硬编码颜色改为 CSS 变量引用

### 7.3 Temurin JDK 25 可能尚未发布
- `environments.js` 中定义了 Temurin JDK 25 的安装命令
- Temurin 25 预计 2025 年 9 月 GA，Chocolatey 包可能尚不存在
- 影响：用户选择 JDK 25 时可能安装失败
- 建议：后续确认 Chocolatey 包是否可用，或暂时隐藏该选项
